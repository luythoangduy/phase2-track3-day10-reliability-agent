# Day 10 Reliability Final Report

## 1. Architecture summary

The gateway handles each request with a cache-first, provider-fallback pipeline. Cache hits return immediately at zero provider cost. Cache misses are routed through a per-provider circuit breaker, starting with the primary provider and falling back to backup providers when a provider fails or its circuit is open. If every provider path fails, the gateway returns a static degraded response.

```text
User Request
    |
    v
[ReliabilityGateway]
    |
    +--> [ResponseCache or SharedRedisCache] -- hit --> cached response
    |
    +--> [CircuitBreaker: primary] -- allowed --> primary provider
    |
    +--> [CircuitBreaker: backup]  -- allowed --> backup provider
    |
    +--> static fallback response
```

## 6. Redis shared cache

In-memory cache is per-process, so two gateway instances cannot share warm responses. `SharedRedisCache` stores query/response pairs in Redis hashes with TTL, making cache state visible across instances and suitable for multi-instance deployments.

## 8. Failure analysis

Remaining weakness: circuit breaker state is still in-process. In a horizontally scaled deployment, one instance can learn that a provider is failing while another instance keeps sending traffic until its local breaker opens.

Production fix: store circuit counters and state in Redis with atomic operations, or publish breaker transitions through a shared control plane. Add per-provider rolling-window metrics so state is based on recent error rate rather than simple consecutive failures.
